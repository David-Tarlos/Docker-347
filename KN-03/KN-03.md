## A) Eigenes Netzwerk

### 1. Aufbau des Zielsystems

Das Zielsystem besteht aus vier `busybox`-Containern, verteilt auf zwei Netzwerke:

```
default bridge (172.17.0.0/16)      tbz (172.18.0.0/16)
├── busybox1                        ├── busybox3
└── busybox2                        └── busybox4
```

#### tbz-Netzwerk anlegen

Das user-defined Netzwerk `tbz` wird mit dem vorgegebenen Subnet `172.18.0.0/16` erstellt. Das default bridge Netzwerk existiert bereits und ist nicht veränderbar.

```bash
docker network create --subnet 172.18.0.0/16 tbz
docker network ls
```

<img width="880" height="476" alt="01_network_create" src="https://github.com/user-attachments/assets/1ade38f9-f4f2-407b-9889-e3e6a60c7f22" />

*Vor dem Befehl existieren nur die Standard-Netzwerke (`bridge`, `host`, `none`) sowie ein älteres Compose-Netz (`minecraft-server_default`). Nach dem Befehl ist `tbz` in der Liste sichtbar – mit Driver `bridge` und Scope `local`.*

#### Vier busybox-Container starten

busybox1 und busybox2 landen ohne `--network`-Flag automatisch im default bridge. busybox3 und busybox4 werden explizit dem `tbz`-Netz zugewiesen. Das Argument `sleep infinity` hält die Container am Leben, damit später interaktive Sessions möglich sind.

```bash
docker run -dit --name busybox1 busybox sleep infinity
docker run -dit --name busybox2 busybox sleep infinity
docker run -dit --name busybox3 --network tbz busybox sleep infinity
docker run -dit --name busybox4 --network tbz busybox sleep infinity
```

**Beobachtungen:**

| Test                         | Ergebnis                  | Erklärung |
|------------------------------|---------------------------|-----------|
| `ifconfig`                   | `eth0: 172.17.0.2`        | IP von busybox1. |
| `route -n` Default-Gateway   | `172.17.0.1`              | Host-Interface im default bridge. **Gleicher Gateway wie busybox2** (beide im selben Netz). |

| `ping 172.17.0.3` (busybox2) | `0% packet loss` ✅       | Gleiches Netz → direkte Erreichbarkeit über IP. |
| `ping 172.18.0.2` (busybox3) | `100% packet loss` ❌     | Anderes Netz → durch Docker isoliert, keine Pakete kommen durch. |

### 5. Gemeinsamkeiten, Unterschiede und Schlussfolgerung

**Gemeinsamkeiten der beiden Netzwerke:**

- Beide sind **Bridge-Netzwerke**. Jeder Container bekommt automatisch eine IP aus dem jeweiligen Subnet zugewiesen.
- In **beiden** Netzen können Container, die im **selben** Netz liegen, sich gegenseitig per IP-Adresse erreichen (ping per IP funktioniert immer im eigenen Netz).
- Der Default-Gateway jedes Containers ist der **Host** – jeweils mit der ersten IP des Subnets (`172.17.0.1` bzw. `172.18.0.1`). Der Host ist also gleichzeitig in beiden Netzwerken Mitglied.

**Unterschiede:**

| Eigenschaft                       | default bridge                          | user-defined bridge (tbz)            |
|-----------------------------------|------------------------------------------|--------------------------------------|
| DNS-Auflösung von Container-Namen | **nein** – nur per IP möglich           | **ja** – `ping busybox4` funktioniert |
| Subnet/IP-Range                   | von Docker vorgegeben (`172.17.0.0/16`) | beim Erstellen selbst wählbar        |
| Konfigurierbar zur Laufzeit       | nur eingeschränkt                       | flexibler                            |
| Isolation                         | Container darin sind unter sich         | Container darin sind unter sich, **getrennt von anderen Netzen** |

**Wie kommen die Zustände zustande?**

Docker verwaltet pro Bridge-Netzwerk eine eigene virtuelle Linux-Bridge (`docker0` für default, `br-<id>` für tbz). Jeder Container bekommt ein virtuelles Netzwerk-Interface, das mit der entsprechenden Bridge verbunden ist. Pakete innerhalb derselben Bridge werden direkt weitergeleitet, **zwischen verschiedenen Bridges blockiert Docker per iptables-Regel** den Verkehr standardmässig. Daher das `100% packet loss` bei Pings über die Netzgrenze hinweg.

Die DNS-Auflösung in user-defined Netzwerken liefert ein eingebauter Docker-DNS-Server (`127.0.0.11`), der die Container-Namen kennt. Im default bridge ist dieser Mechanismus nicht aktiv – dort müsste man entweder mit IPs oder mit dem (deprecated) `--link`-Flag arbeiten.

**Schlussfolgerung:**

Wer Container braucht, die sicher und komfortabel miteinander kommunizieren sollen, **verwendet ein user-defined Netzwerk**. Vorteile:

1. **Container-Namen statt IPs:** Stabil, auch wenn IPs sich beim Neustart ändern.
2. **Isolation gegen aussen:** Container ausserhalb des Netzes haben keinen Zugriff.
3. **Klare Architektur:** Jede Anwendung kann ihr eigenes Netz haben (z.B. eines pro Projekt).

Die Docker-Dokumentation empfiehlt user-defined Netzwerke deshalb ausdrücklich – das default bridge gilt als Legacy.

## B) Reflexion und Verbesserung von KN02

In KN02 wurde ein Web-Container (PHP/Apache) und ein DB-Container (MariaDB) aufgebaut. Die Web-App ruft die DB an, um eine Verbindung zu prüfen (`db.php`).

### B.1 In welchem Netzwerk befanden sich Web- und DB-Container?

Beide Container liefen im **default bridge** Netzwerk. Der Web-Container wurde zusätzlich über das damals gängige (aber inzwischen deprecated) `--link`-Flag mit dem DB-Container verknüpft:

```bash
docker run -d --name kn02b-db  -p 3306:3306 leonwul/m347:kn02b-db
docker run -d --name kn02b-web -p 80:80 --link kn02b-db leonwul/m347:kn02b-web
```

### B.2 Weshalb funktionierte die Verbindung?

Die Aufgabenstellung fragt explizit nach einer Verbindung über die **IP-Adresse**. In unserer KN02-Umsetzung wurde **nicht** direkt eine fixe IP eingetragen, sondern stattdessen der Container-Name `kn02b-db` mit Hilfe des `--link`-Flags verwendet. Der Mechanismus dahinter ist aber funktional vergleichbar mit einer fixen IP – deshalb behandeln wir beide Fälle gemeinsam:

- **Variante "fixe IP eintragen":** Wenn man im PHP-Code direkt `$host = '172.17.0.3'` setzt, funktioniert die Verbindung, weil beide Container im selben default bridge liegen und somit per IP erreichbar sind (vgl. das Pingen per IP im Teil A). Innerhalb desselben Bridge-Netzes routet Docker den Verkehr direkt.
- **Variante `--link` (unser Fall):** `--link` schreibt beim Container-Start einen Eintrag in die `/etc/hosts` des Web-Containers, der den Namen `kn02b-db` auf die aktuelle IP des DB-Containers auflöst. Das funktioniert nur, weil beide im gleichen Netzwerk liegen und Docker die zum Startzeitpunkt aktuelle IP in die Hosts-Datei einträgt.

Beide Varianten setzen voraus, dass die Container im **gleichen Netzwerk** liegen.

### B.3 Weshalb ist eine direkt eingetragene Container-IP nicht ideal?

Mehrere Gründe sprechen klar gegen das Hardcoden einer IP (und in abgeschwächter Form auch gegen `--link`):

1. **IPs ändern sich:** Beim Stoppen/Neustarten eines Containers kann Docker eine andere IP aus dem Pool vergeben. Die App findet die DB dann nicht mehr und ist kaputt.
2. **Nicht portabel:** Bei einem Deployment auf einem anderen Rechner haben die Container fast garantiert andere IPs als auf dem Entwicklungssystem.
3. **Schwer wartbar:** Die IP ist im Code/Config versteckt. Beim Ändern müsste man alle Stellen im Code finden und anpassen.
4. **`--link` ist deprecated:** Docker selbst empfiehlt, `--link` nicht mehr zu verwenden, sondern auf user-defined Netzwerke umzusteigen.
5. **Sicherheits-Aspekt:** Die Container liegen im default bridge, in dem alle anderen "namenlosen" Container ebenfalls landen. Es gibt keine saubere Trennung von Anwendungen voneinander. Zusätzlich war der DB-Port (`-p 3306:3306`) nach aussen exponiert, obwohl die DB nur vom Web-Container intern erreichbar sein muss – ein Verstoss gegen das **Principle of Least Privileges**.

### B.4 Verbesserungsvorschlag

Statt im default bridge zu arbeiten und `--link` zu verwenden, legen wir ein **eigenes user-defined Netzwerk** (`kn02-net`) an und packen beide Container hinein. Damit gewinnen wir:

- **DNS-Auflösung über Container-Namen:** Die `db.php` kann weiterhin `$host = 'kn02b-db'` verwenden, ohne dass `--link` nötig ist. Docker löst den Namen automatisch zur aktuellen IP auf.
- **Isolation:** Container ausserhalb von `kn02-net` haben keinen Zugriff auf die DB.
- **Port 3306 wird nicht mehr nach aussen gemappt:** Die DB ist nur noch innerhalb von `kn02-net` erreichbar (Principle of Least Privileges).
- **Bessere Wartbarkeit:** Wenn DB oder Web neu starten und neue IPs bekommen, funktioniert die Anwendung weiterhin.

### B.5 Umsetzung der Verbesserung

**1. Eigenes Netzwerk erstellen:**

```bash
docker network create kn02-net
```

**2. Images neu builden:**

Die Dockerfiles aus KN02 bleiben unverändert. Auch die `db.php` muss nicht angepasst werden – `$host = 'kn02b-db'` funktioniert weiterhin, da der Container-Name jetzt über den Docker-DNS aufgelöst wird (statt über den `--link`-Trick mit `/etc/hosts`).

```bash
docker build -f Dockerfile.kn02b-db  -t kn02b-db  .
docker build -f Dockerfile.kn02b-web -t kn02b-web .
```

**3. Container starten – beide im selben user-defined Netz, DB ohne Port-Mapping nach aussen:**

```bash
docker run -d --name kn02b-db  --network kn02-net          kn02b-db
docker run -d --name kn02b-web --network kn02-net -p 80:80 kn02b-web
```

**Wichtigste Änderungen gegenüber KN02:**

- Beide Container haben `--network kn02-net` statt `--link`.
- Der DB-Container hat **kein** `-p 3306:3306` mehr → DB ist nur intern erreichbar.

<img width="911" height="891" alt="09_kn02_build_run" src="https://github.com/user-attachments/assets/b9042070-330c-436c-b3d1-d42068d5b0b6" />

*Build der beiden Images, anschliessend Start der Container im `kn02-net`. Im `docker ps`-Output ist die Sicherheits-Verbesserung deutlich sichtbar.*

Auszug aus dem `docker ps`-Output:

```
CONTAINER ID   IMAGE       ...   PORTS                NAMES
b402674103ae   kn02b-web   ...   0.0.0.0:80->80/tcp   kn02b-web
99a7dab7d8b2   kn02b-db    ...   3306/tcp             kn02b-db
```

Der Web-Container hat ein Port-Mapping (`0.0.0.0:80->80/tcp`) – ist also vom Host und damit vom Browser erreichbar. Der DB-Container zeigt nur `3306/tcp` ohne `0.0.0.0:`-Prefix – der Port ist **nur innerhalb** des Container-Netzes offen, **nicht** vom Host.
