### Erklärungen zu den Container-Befehlen

#### Teil 1 – `docker run -d ubuntu`

> Erstellen und starten Sie einen Container mit dem Befehl `docker run -d`.
> Kommentieren Sie in 3–5 Sätzen, was mit dem Container geschieht.
> Wurde das Image automatisch heruntergeladen? Konnte es starten?

**Antwort:** Das Ubuntu-Image wurde automatisch von Docker Hub
heruntergeladen, da es lokal noch nicht vorhanden war. Der Container
wurde erstellt und gestartet, hat sich aber sofort wieder beendet
(Status `Exited`). Grund dafür ist, dass das Ubuntu-Image keinen
langlaufenden Hauptprozess besitzt – anders als z. B. nginx, das einen
Webserver-Prozess mitbringt. Ein Container lebt nur so lange wie sein
Hauptprozess; ohne aktiven Prozess endet er sofort. Das Flag `-d`
(detached) macht hier keinen Unterschied, weil es gar nichts gibt, was
im Hintergrund laufen könnte.

#### Teil 2 – `docker run -it ubuntu`

> Erstellen und starten Sie einen Container mit dem Befehl `docker run -it`.
> Kommentieren Sie in 3–5 Sätzen, was nun geschieht.

**Antwort:** Mit den Flags `-i` (interactive) und `-t` (TTY) wird beim
Start des Containers eine interaktive Shell geöffnet. Man landet direkt
im Bash-Prompt des Ubuntu-Containers (z. B. `root@abc123:/#`) und kann
Linux-Befehle ausführen, als wäre man auf einem normalen Ubuntu-System.
Die Bash ist nun der Hauptprozess des Containers und hält ihn am Leben,
solange die Shell offen bleibt. Mit dem Befehl `exit` wird die Shell
beendet – und damit auch der Container, weil der Hauptprozess endet.
Das zeigt, dass Ubuntu nur dann sinnvoll als Container nutzbar ist, wenn
man ihm aktiv einen Prozess (wie eine Shell) zur Verfügung stellt.

## D) Privates Repository (20%)

Ziel: Die öffentlichen Images `nginx` und `mariadb` in ein eigenes
Docker-Hub-Repository übertragen und dabei den Unterschied zwischen
Image, Repository und Tag verstehen.

### Ausgeführte Befehle

```bash
# 1. Offizielles nginx-Image (nochmals) herunterladen
docker pull nginx

# 2. Am Docker-Hub anmelden
docker login

# 3. Lokales nginx-Image für das eigene Repository taggen
docker tag nginx:latest david-tarlos/docker-347:nginx

# 4. Getaggtes Image hochladen
docker push david-tarlos/docker-347:nginx

# 5. Offizielles mariadb-Image herunterladen
docker pull mariadb

# 6. mariadb-Image für das eigene Repository taggen
docker tag mariadb:latest david-tarlos/docker-347:mariadb

# 7. mariadb-Image hochladen
docker push david-tarlos/docker-347:mariadb
```

> `david-tarlos/docker-347` ist hier der Repository-Pfad
> (`Benutzername/reponame`). Bei der eigenen Abgabe entsprechend
> den eigenen Docker-Hub-Benutzernamen einsetzen.

### Erklärung der Befehle

#### Was ist ein **Tag**?

Ein Tag ist eine Beschriftung (Label), die innerhalb eines Repositories
eine bestimmte Version oder Variante eines Images identifiziert. Das
vollständige Image-Bezeichnungsschema lautet:

```
[registry/]benutzername/repository:tag
```

Beispiele:

| Beispiel | Bedeutung |
|---|---|
| `nginx:latest` | aktuellster Stand des offiziellen nginx-Repos |
| `nginx:1.25-alpine` | konkrete Version + Variante (Alpine Linux) |
| `david-tarlos/docker-347:nginx` | im eigenen Repo wird `nginx` als Tag verwendet, um die Applikation zu kennzeichnen |

Tags dienen klassischerweise der **Versionierung**, können aber – wie in
dieser Aufgabe – auch genutzt werden, um **mehrere Applikationen in einem
Repository zusammenzufassen**. Ohne explizit angegebenen Tag verwendet
Docker automatisch `:latest`.

#### `docker tag nginx:latest david-tarlos/docker-347:nginx`

Dieser Befehl erstellt **keinen** neuen Image-Inhalt – er vergibt lediglich
einen zusätzlichen Namen (Alias) für ein bereits vorhandenes Image. Das
Quell-Image `nginx:latest` und das Ziel-Image
`david-tarlos/docker-347:nginx` zeigen anschliessend auf dieselbe
Image-ID; die Layer werden **nicht** kopiert. Erst durch das Taggen mit
dem eigenen Benutzernamen kann das Image später auf Docker Hub in das
eigene Repository gepusht werden, denn Docker Hub akzeptiert Pushes nur
unter dem eigenen Namespace.

#### `docker push david-tarlos/docker-347:nginx`

Dieser Befehl lädt das lokal getaggte Image auf den Docker-Hub-Server
hoch. Docker überträgt dabei nur jene Layer, die im Ziel-Repository noch
nicht vorhanden sind (Layer-Caching auf Registry-Ebene). Voraussetzung
ist eine vorherige Anmeldung mit `docker login`. Nach dem Push ist das
Image öffentlich (oder privat, je nach Repo-Einstellung) unter
`hub.docker.com/r/david-tarlos/docker-347` mit dem Tag `nginx` sichtbar.

#### `docker pull mariadb` + `docker tag …` + `docker push …`

Genau dasselbe Vorgehen wie für nginx, nur mit dem mariadb-Image. Danach
enthält das eigene Repository `david-tarlos/docker-347` **zwei Tags**:
`nginx` und `mariadb` – beide Applikationen sind im selben Repository
zusammengefasst.
