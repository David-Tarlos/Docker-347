# Docker-347

## KN05 – A) Bind Mounts (40%)

### Ziel
Beweisen, dass ein Bind Mount den Speicher zwischen Host (Windows) und Container teilt:
Eine auf dem Host geänderte Datei ist sofort im Container sichtbar – ohne Neustart.
Simuliert den Entwickler-Alltag: Code lokal ändern, direkt im Container testen.

### Liste der Befehle
```bash
# 1. Container mit Bind Mount starten
docker run -d --name kn05-bind -v C:\m347\bindmount:/mnt/host nginx:latest

# 2. In den laufenden Container wechseln
docker exec -it kn05-bind bash

# 3. Im Container: Skript finden und ausführen
cd /mnt/host
ls
bash script.sh

# 4. Nach der Änderung auf dem Host: erneut ausführen (kein Neustart!)
bash script.sh
```

Der Bind Mount `-v C:\m347\bindmount:/mnt/host` verbindet den Host-Ordner
(links) mit dem Pfad `/mnt/host` im Container (rechts). Beide zeigen auf
dieselben Dateien – es ist keine Kopie.

## KN05 – B) Volumes (30%)

### Ziel
Beweisen, dass sich zwei Container dasselbe Named Volume teilen können:
Schreibt ein Container in eine Datei, kann der andere sie sofort lesen – und
umgekehrt. Im Gegensatz zum Bind Mount liegt der Speicher nicht in einem
Host-Ordner, sondern wird von Docker selbst verwaltet.

### Liste der Befehle
```bash
# 1. Named Volume erstellen
docker volume create kn05-vol

# 2. Volume anzeigen / prüfen
docker volume ls

# 3. Zwei Container mit demselben Volume starten
docker run -d --name kn05-c1 -v kn05-vol:/data nginx:latest
docker run -d --name kn05-c2 -v kn05-vol:/data nginx:latest

# 4. Container 1: in geteilte Datei schreiben
docker exec -it kn05-c1 bash
echo "Nachricht von Container 1" >> /data/shared.txt
cat /data/shared.txt

# 5. Container 2: lesen und dazuschreiben
docker exec -it kn05-c2 bash
cat /data/shared.txt
echo "Antwort von Container 2" >> /data/shared.txt
cat /data/shared.txt

# 6. Container 1: Antwort von Container 2 lesen
cat /data/shared.txt
```

Bei einem Named Volume steht links vom Doppelpunkt der **Volume-Name**
(`kn05-vol`) statt eines Host-Pfads. Beide Container hängen dasselbe Volume
unter `/data` ein und greifen so auf denselben Speicher zu.

## KN05 – C) Speicher mit docker compose (30%)

### Ziel
Alle drei Speichertypen deklarativ in einer `docker-compose.yml` definieren:
- **Container 1 (`web1`)** erhält alle drei Typen: named volume (Long Syntax),
  bind mount (Long Syntax) und tmpfs.
- **Container 2 (`web2`)** erhält nur das named volume, bewusst in Short Syntax.

Die drei Typen kurz erklärt:
- **named volume** – von Docker verwaltet, überlebt einen Container-Neustart
- **bind mount** – Ordner vom Host (wie in Teil A)
- **tmpfs** – liegt nur im RAM, ist weg sobald der Container stoppt

### docker-compose.yml
```yaml
services:
  # Container 1 - bekommt alle drei Speichertypen
  web1:
    image: nginx:latest
    container_name: m347-kn05c-web1
    volumes:
      # Named Volume in LONG Syntax
      - type: volume
        source: kn05c-vol
        target: /data
      # Bind Mount in LONG Syntax
      - type: bind
        source: ./bindmount
        target: /mnt/host
    tmpfs:
      # tmpfs Mount - liegt nur im RAM
      - /app/cache

  # Container 2 - bekommt nur das Named Volume
  web2:
    image: nginx:latest
    container_name: m347-kn05c-web2
    volumes:
      # Named Volume in SHORT Syntax
      - kn05c-vol:/data

# Top Level Element fuer das Named Volume
volumes:
  kn05c-vol:
```

Zentrale Punkte: Das **Top-Level-Element `volumes:`** (unten) definiert das
named volume `kn05c-vol`. Beide Container nutzen dasselbe Volume – `web1` per
**Long Syntax** (`type: volume`, mehrzeilig), `web2` per **Short Syntax**
(`kn05c-vol:/data`, einzeilig). Der bind mount (`type: bind`) und der tmpfs
mount (`/app/cache`) hängen nur an `web1`.

### Liste der Befehle
```bash
# 1. Beide Container starten
docker compose up -d

# 2. Status prüfen
docker compose ps

# 3. mount im ersten Container (alle drei Typen)
docker exec -it m347-kn05c-web1 bash
mount | grep -E "/data|/mnt/host|/app/cache"

# 4. mount im zweiten Container (nur named volume)
docker exec -it m347-kn05c-web2 bash
mount | grep /data
```

### Interpretation des `mount`-Auszugs (Container 1)
| Zeile im Auszug | Speichertyp |
|-----------------|-------------|
| `/dev/sdd on /data type ext4` | **named volume** (`kn05c-vol`), von Docker als echtes Dateisystem eingehängt |
| `tmpfs on /app/cache type tmpfs` | **tmpfs mount**, liegt nur im RAM |
| `C:\ on /mnt/host type 9p` | **bind mount**, der Windows-Ordner (9p ist das Protokoll, über das Docker Desktop Host-Ordner unter Windows durchreicht) |

### Fazit
Container 1 enthält alle drei Speichertypen (Bild 3) – named volume, bind mount
und tmpfs. Container 2 enthält wie gewünscht nur das named volume (Bild 4),
angebunden über die Short Syntax. Damit sind alle Vorgaben der Aufgabe erfüllt:
beide Syntax-Varianten für das geteilte Volume sowie die zusätzlichen
Speichertypen am ersten Container.
