## Block 1 — Docker

### 1.1 VM vs Container

> **Lernziel:** "Ich kann den Unterschied zwischen einer virtuellen Maschine (VM) und einem Container erklären und die Vorteile von Container-Isolierung benennen."

**Kerngedanke:** Eine **VM** hat einen eigenen kompletten OS-Kernel (über einen Hypervisor). Ein **Container** teilt sich den OS-Kernel des Hosts und isoliert nur den User-Space (Prozesse, Filesystem, Netzwerk).

```
┌─────────────────────────────┐       ┌─────────────────────────────┐
│  VM 1   │  VM 2   │  VM 3   │       │ Container 1│ Container 2│..  │
│ OS A    │ OS B    │ OS C    │       │ App + libs │ App + libs │    │
│ Kernel A│ Kernel B│ Kernel C│       │   (User-Space only)            │
├─────────────────────────────┤       ├─────────────────────────────┤
│       Hypervisor            │       │   Container Engine (Docker) │
├─────────────────────────────┤       ├─────────────────────────────┤
│       Host-OS + Kernel      │       │       Host-OS + Kernel      │
├─────────────────────────────┤       ├─────────────────────────────┤
│         Hardware            │       │         Hardware            │
└─────────────────────────────┘       └─────────────────────────────┘
```

**Vorteile von Containern gegenüber VMs:**
- **Kleiner:** Image im MB-Bereich vs VM-Image in GB
- **Schneller:** Start in Sekunden vs Minuten
- **Effizienter:** Weniger CPU/RAM-Overhead (kein eigenes Kernel)
- **Portabel:** Läuft überall wo eine passende Container-Engine + Kernel-Version vorhanden ist
- **Reproduzierbar:** Image ist immer gleich — keine "auf meinem Rechner geht's"-Probleme

**Nachteile (für die Prüfung gut zu wissen):**
- Schwächere Isolation (alle Container teilen den gleichen Kernel — Kernel-Bug betrifft alle)
- Kein anderes OS möglich (Linux-Container brauchen Linux-Kernel — Windows-Container auf Linux geht nur mit Tricks)

**Praxis-Anker:** Du hast alle KNs mit Containern gemacht — die laufen auf einem einzigen Host-Kernel, viel ressourcenschonender als 4 VMs.

### 1.2 Image vs Container

> **Lernziel:** "Ich kann den Unterschied zwischen einem Docker Image und einem Docker Container erklären."

**Kerngedanke:**
- **Image** = der **Bauplan** (read-only, statisch). Vergleichbar mit einer Klasse in der Programmierung, oder einer ISO-Datei.
- **Container** = die **laufende Instanz** eines Images (read-write Layer drüber). Vergleichbar mit einem Objekt der Klasse, oder einer gerade benutzten VM aus einer ISO.

```
                  docker run
   Image (Bauplan) ────────────► Container 1 (laufend)
                  ────────────► Container 2 (laufend)
                  ────────────► Container 3 (laufend)
```

Aus einem Image können beliebig viele Container parallel gestartet werden. Jeder hat seinen eigenen Schreib-Layer obendrauf — Änderungen in Container 1 sind nicht in Container 2.

**Wichtige Befehle zur Unterscheidung:**
```bash
docker images          # Liste aller Images (Baupläne)
docker ps              # Liste laufender Container
docker ps -a           # Liste aller Container (auch gestoppte)
docker rm <container>  # löscht Container
docker rmi <image>     # löscht Image
```

**Stolperfalle:** `docker rm` löscht NICHT das Image — nur den Container. Für das Image brauchst du `docker rmi`.

**Praxis-Anker:** In KN-04 hattest du das Image `leonwul/m347:kn02b-web` und davon den Container `m347-kn04b-web`.

### 1.3 CLI-Befehle

> **Lernziel:** "Ich kann Container-Images mittels CLI-Befehlen (build, tag, push) verwalten und lokale Instanzen starten/stoppen"

**Die wichtigsten Befehle mit Beispielen:**

```bash
# IMAGE BAUEN aus Dockerfile (im aktuellen Verzeichnis)
docker build -t leonwul/myapp:v1 -f Dockerfile .
#         │              │       │           │
#         │              │       │           └─ Build Context (Pfad mit Dockerfile + COPY-Quellen)
#         │              │       └─ Welches Dockerfile (Default: ./Dockerfile)
#         │              └─ Tag im Format namespace/image:version
#         └─ -t = "tag" (vergibt Namen)

# IMAGE TAG NACHTRÄGLICH SETZEN / UMBENENNEN
docker tag leonwul/myapp:v1 leonwul/myapp:latest

# IMAGE IN REGISTRY HOCHLADEN
docker push leonwul/myapp:v1
# Voraussetzung: vorher `docker login` (für DockerHub: deine Login-Daten)

# IMAGE AUS REGISTRY HERUNTERLADEN (wird bei docker run automatisch gemacht falls fehlt)
docker pull nginx:latest

# CONTAINER STARTEN
docker run -d -p 8080:80 --name web leonwul/myapp:v1
#          │  │           │         │
#          │  │           │         └─ welches Image
#          │  │           └─ Name des Containers (sonst zufälliger Name)
#          │  └─ Port-Mapping: Host:Container (Anfrage auf Host-8080 geht zu Container-80)
#          └─ -d = "detached" (im Hintergrund laufen lassen)
# Mit -it stattdessen interaktiv: docker run -it ubuntu bash

# CONTAINER ANZEIGEN
docker ps              # laufende
docker ps -a           # auch gestoppte

# CONTAINER STOPPEN + LÖSCHEN
docker stop web        # sendet SIGTERM, beendet sauber
docker rm web          # entfernt gestoppten Container

# IN LAUFENDEM CONTAINER EINEN BEFEHL AUSFÜHREN
docker exec -it web bash    # öffnet Shell im Container

# LOGS ANSCHAUEN
docker logs web
docker logs -f web     # -f = follow (live mit)
```

**Was kann gefragt werden:**
- Was bedeutet `-d` (detached), `-it` (interactive+tty), `-p` (port-mapping)?
- Was passiert wenn man kein `--name` setzt? → Docker generiert einen zufälligen Namen wie `friendly_einstein`.
- Reihenfolge `docker stop` dann `docker rm` — warum nicht direkt `docker rm`? → `docker rm` funktioniert nur bei gestoppten Containern (ausser man nimmt `-f` für force).

**Praxis-Anker:** Alle 4 Images in KN-08 hast du mit `docker build` gebaut, `docker push` zu DockerHub geschoben und mit `docker run` lokal getestet.

### 1.4 Dockerfile-Anweisungen

> **Lernziel:** "Ich bin in der Lage die folgenden Anweisungen für Dockerfiles zu verwenden oder zu erklären: FROM, COPY, WORKDIR, RUN, EXPOSE, ENV, CMD"

**Die 7 wichtigsten Anweisungen mit Erklärung + Beispiel:**

```dockerfile
# 1. FROM — Basis-Image (immer die erste Anweisung)
FROM node:20-alpine
# Lädt das offizielle Node.js-20-Image (Alpine-Linux-Variante) als Startpunkt.

# 2. WORKDIR — Arbeitsverzeichnis im Container setzen
WORKDIR /app
# Alle folgenden RUN/COPY/CMD-Anweisungen werden in /app ausgeführt.
# Wenn /app nicht existiert, wird es erstellt.

# 3. COPY — Dateien vom Host (Build-Context) ins Image kopieren
COPY package*.json ./
COPY src/ ./src/
# package*.json matched package.json + package-lock.json.
# ./ ist relativ zu WORKDIR, also /app im Image.

# 4. RUN — Befehl im Image ausführen (zur Build-Zeit, einmalig)
RUN npm install --omit=dev
# Wird beim Image-Bauen ausgeführt, das Ergebnis wird in einer neuen Layer abgelegt.
# Nicht zu verwechseln mit CMD (wird beim Container-Start ausgeführt).

# 5. ENV — Umgebungsvariable setzen (gilt im Image + Container)
ENV NODE_ENV=production
ENV PORT=8080
# Können in RUN-Anweisungen und im laufenden Container gelesen werden.

# 6. EXPOSE — dokumentiert welchen Port der Container hört
EXPOSE 8080
# WICHTIG: EXPOSE öffnet NICHT den Port. Es ist nur Dokumentation für Leute, die das Image lesen.
# Tatsächlich erreichbar wird der Port erst durch `docker run -p ...`.

# 7. CMD — Default-Befehl beim Container-Start
CMD ["node", "index.js"]
# Wird ausgeführt sobald der Container hochfährt.
# Im JSON-Array-Format (empfohlen) statt Shell-Form.
# Kann beim `docker run` überschrieben werden: `docker run myimage other-command`
```

**Bonus-Anweisungen (gut für die Prüfung zu kennen):**
- `ADD` — wie COPY, kann zusätzlich URLs herunterladen und tar-Archive entpacken (modern: lieber COPY)
- `ENTRYPOINT` — wie CMD, aber kann nicht beim docker run überschrieben werden (nur Argumente werden angehängt)
- `USER` — wechselt zum angegebenen User (nicht root) für die folgenden Anweisungen
- `VOLUME` — deklariert einen Mountpoint für Volumes
- `HEALTHCHECK` — periodischer Test ob Container gesund ist
- `MAINTAINER` — Metadata (heute eher `LABEL maintainer="..."`)

**Was kann gefragt werden:**
- Unterschied RUN vs CMD? → RUN = Build-Zeit (einmalig, baut Image-Layer); CMD = Container-Start (jedes Mal).
- Unterschied CMD vs ENTRYPOINT? → CMD kann beim run überschrieben werden, ENTRYPOINT nicht (sondern nur erweitert).
- Was macht WORKDIR genau? → Setzt das aktuelle Verzeichnis für folgende Anweisungen UND erstellt das Verzeichnis falls nicht vorhanden.
- Öffnet EXPOSE wirklich den Port? → NEIN, nur Dokumentation. Port wird durch `docker run -p` veröffentlicht.

**Praxis-Anker:** Dein KN-02 Dockerfile war:
```dockerfile
FROM php:8.0-apache
RUN docker-php-ext-install mysqli
COPY info.php /var/www/html/
COPY db.php /var/www/html/
EXPOSE 80
```

### 1.5 Layers

> **Lernziel:** "Ich kann erklären was ein Layer in einem Container ist und wie Layers erstellt werden."

**Kerngedanke:** Ein Docker-Image besteht aus **gestapelten Schichten** (Layers). **Jede Anweisung im Dockerfile = eine neue Layer.**

```
┌──────────────────────────────────┐  ← Read-Write Layer (vom Container)
├──────────────────────────────────┤  ← Layer 5: CMD ["node", "index.js"]
├──────────────────────────────────┤  ← Layer 4: COPY src/ ./src/
├──────────────────────────────────┤  ← Layer 3: RUN npm install
├──────────────────────────────────┤  ← Layer 2: COPY package*.json ./
├──────────────────────────────────┤  ← Layer 1: WORKDIR /app
├──────────────────────────────────┤  ← Layer 0: FROM node:20-alpine
```

**Wie entsteht eine Layer technisch?**
- Beim Build wird ein temporärer Container vom vorherigen Layer gestartet
- Die Dockerfile-Anweisung wird darin ausgeführt
- Die Änderungen am Filesystem werden als neue Layer gespeichert
- Der temporäre Container wird gelöscht
- Die neue Layer wird im Image-Stack obendrauf gelegt

**Warum sind Layers so wichtig?** → **Caching!**
- Wenn nichts geändert wurde, nutzt Docker beim nächsten Build die alte Layer aus dem Cache
- Build wird viel schneller
- ABER: ändert sich eine Layer, müssen ALLE darüberliegenden Layers neu gebaut werden

**Best Practice:** Häufig ändernde Dinge nach unten, selten ändernde nach oben.
```dockerfile
FROM node:20-alpine         # ändert sich nie → ganz oben
WORKDIR /app                 # ändert sich nie
COPY package*.json ./        # ändert sich selten (nur bei neuen Dependencies)
RUN npm install              # ändert sich nur wenn package.json geändert
COPY . .                     # ändert sich BEI JEDER Code-Änderung → ganz unten
CMD ["node", "index.js"]
```
Wenn du nur Source-Code änderst (kein neues package.json), spart Docker dir den teuren `npm install` durch Layer-Caching.

**Praxis-Anker:** Beim Multistage-Build in KN-08 hast du gesehen wie schnell der zweite Build war — Layer-Cache hat gegriffen.

### 1.6 Volumes + Netzwerke

> **Lernziel:** "Ich kann Volumes und Netzwerkkonfigurationen in Docker anwenden, um Datenpersistenz und Kommunikation zwischen Containern sicherzustellen."

**Zwei Probleme die Container lösen müssen:**

**Problem A — Persistenz:** Wenn ein Container stirbt, sind seine Daten weg (read-write Layer wird gelöscht). DB-Daten sind dann futsch. → Lösung: **Volumes**

```bash
# Bei docker run einen Volume mounten:
docker run -d -v mydata:/var/lib/mysql --name db mysql

# Im docker-compose:
services:
  db:
    image: mysql
    volumes:
      - mydata:/var/lib/mysql
volumes:
  mydata:
```
Der Pfad `/var/lib/mysql` im Container wird auf ein **Docker-managed Volume** namens `mydata` umgeleitet. Container kann sterben — die Daten in `mydata` bleiben.

**Problem B — Kommunikation:** Container in Default-Konfig können sich nicht über Container-Namen erreichen. → Lösung: **eigenes Netzwerk** (user-defined bridge)

```bash
# Eigenes Netzwerk erstellen
docker network create my-net

# Container im Netz starten
docker run -d --name db --network my-net mysql
docker run -d --name web --network my-net myapp

# Jetzt kann "web" mit "db" über den Namen reden:
# Im web-Container: mysql://db:3306 funktioniert
```
Im selben user-defined Netzwerk fungiert der Container-Name automatisch als DNS-Hostname.

**Praxis-Anker:**
- Volumes — KN-05 hattest du bind mounts + named volumes für Storage-Demos
- Netzwerke — In KN-03 / KN-04 hast du Container über Custom-Networks verbunden, in KN-07 über Compose-Networks

### 1.7 Volume-Typen

> **Lernziel:** "Ich kenne die Unterschiede, Anwendungszwecke und Eigenheiten der Volume-Typen erklären"

**Docker kennt 3 Arten:**

| Typ | Wo gespeichert | Verwaltung | Use-Case | Persistent? |
|---|---|---|---|---|
| **bind mount** | beliebiger Pfad auf Host | du selbst | Code mit Container sharen während Entwicklung | ja (auf Host) |
| **volume** | Docker-managed Verzeichnis (`/var/lib/docker/volumes/`) | Docker | DB-Daten, Production-Persistenz | ja (Docker-managed) |
| **tmpfs** | RAM des Hosts | flüchtig | temporäre Daten, sensitive Secrets | NEIN (weg bei Stop) |

**Bind Mount — Dev-Workflow:**
```bash
docker run -d -v "$(pwd)/src":/app/src node:20 npm start
```
Code-Datei im `src/`-Ordner ändern → Container sieht die Änderung sofort. Praktisch für Hot-Reload während Development.

**Volume — Production-Persistenz:**
```bash
docker volume create mysql-data
docker run -d -v mysql-data:/var/lib/mysql mysql
```
Docker entscheidet wo physisch gespeichert. Du arbeitest nur mit dem Namen `mysql-data`. Portabler.

**tmpfs — temporäre Daten:**
```bash
docker run --tmpfs /tmp/cache -d myapp
```
`/tmp/cache` im Container ist im RAM des Hosts. Stop = alles weg. Gut für sensitive Werte die nie auf Disk landen sollen, oder für Speed.

**Was kann gefragt werden:**
- Welcher Typ ist Docker-managed? → volume
- Welcher Typ überlebt einen Container-Restart NICHT? → tmpfs
- Welcher ist am besten für DB-Daten in Production? → volume (mit Docker-Verwaltung) oder bind mount auf ein dediziertes Volume
- Was passiert mit volume wenn ich den Container lösche? → Volume bleibt, kann von anderem Container wieder genutzt werden
- Wie löscht man ein nicht mehr benötigtes Volume? → `docker volume rm <name>` oder `docker volume prune` für alle ungenutzten

**Praxis-Anker:** KN-05 (Storage) hat alle 3 Typen behandelt.

### 1.8 Default-Bridge vs User-defined Bridge

> **Lernziel:** "Ich kenne die Unterschiede zwischen dem Default-Bridge und einem eigenen erstellten Netzwerk und kann sie erklären"

**Wenn du nichts angibst**, hängen alle Container im **default bridge** (`bridge` heisst es). Das ist meistens nicht was du willst.

| Eigenschaft | Default bridge | User-defined bridge |
|---|---|---|
| **DNS via Container-Name** | ❌ NEIN — funktioniert nicht | ✅ JA — `ping mein-db` reicht |
| **Isolation zwischen Container** | alle Container im Default-Bridge sehen sich | Container in unterschiedlichen Netzen sehen sich NICHT |
| **Config-Änderungen zur Laufzeit** | erfordert Docker-Restart | jederzeit möglich |
| **Manuelle Erstellung** | existiert automatisch | `docker network create my-net` |
| **Empfohlen von Docker?** | nein (legacy) | ja |

**Wie ein eigenes Netz erstellen + nutzen:**
```bash
docker network create my-net
docker run -d --name db --network my-net mysql
docker run -d --name web --network my-net myapp

# Jetzt von web-Container aus:
ping db          # ✅ funktioniert (DNS via Container-Name)
```

Mit dem **Default-Bridge** müsstest du die IP-Adresse statt Name nehmen — und die kann sich beim Restart ändern. **Deshalb immer user-defined bridge benutzen.**

**Praxis-Anker:** In KN-04 hast du eigene Netzwerke mit Custom-Subnets erstellt:
```yaml
networks:
  kn04a-net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.10.0.0/16
          ip_range: 172.10.5.0/24
          gateway: 172.10.5.254
```
Das ist eine user-defined bridge mit zusätzlich definiertem IP-Range.

## Block 2 — Container Registry

### 2.1 Was ist eine Container Registry

> **Lernziel:** "Ich kann erklären was eine Container Registry (z. B. GitLab Registry oder Docker Hub) ist und wie Sie sinnvoll verwendet werden kann."

**Definition:** Eine Container Registry ist ein **zentraler Speicherort für Container-Images** — ähnlich wie GitHub für Code, nur eben für Images.

**Bekannte Registries:**
- **Docker Hub** (`hub.docker.com`) — die populärste, öffentliche Default-Registry
- **GitHub Container Registry** (`ghcr.io`)
- **GitLab Container Registry** (`registry.gitlab.com`)
- **AWS ECR**, **Azure ACR**, **Google GCR** — Cloud-Provider Registries
- **MicroK8s Local Registry** — `microk8s enable registry` (port 32000), lokal im Cluster

**Workflow:**
```bash
docker login                            # einmal einloggen
docker build -t leonwul/myapp:v1 .      # bauen mit Registry-Username im Tag
docker push leonwul/myapp:v1            # in Registry hochladen
# auf anderer Maschine:
docker pull leonwul/myapp:v1            # runterladen (oder docker run zieht automatisch)
docker run -d leonwul/myapp:v1
```

**Wozu sinnvoll?**
- **Verteilen** — Image einmal bauen, viele Maschinen können es nutzen
- **Versionierung** — alte Versionen über Tags erreichbar (`:v1`, `:v2`, ...)
- **Reproduzierbarkeit** — gleicher Tag = gleiches Image, immer
- **Automatisierung** — CI/CD: Build pusht automatisch, Deploy zieht automatisch
- **Trennung von Build und Deploy** — Build-Server baut, Deploy-Server zieht

**Public vs Private:**
- **Public:** jeder kann das Image ziehen, kein Login nötig (z.B. `leonwul/m347-kn08`)
- **Private:** Login erforderlich, nur autorisierte Nutzer (typisch in Firmen)

**Praxis-Anker:** In KN-02 hast du dein erstes Image `leonwul/m347:kn02b-web` auf Docker Hub gepusht. In KN-08 hast du einen Public-Repo `leonwul/m347-kn08` mit 4 Tags angelegt.

### 2.2 Semantic Versioning + Image-Tags

> **Lernziel:** "Ich verstehe das Konzept von 'Semantic Versioning' bei Image-Tags (z. B. :latest vs. :1.0.4) und kann dieses in einem Workflow anwenden."

**Tags** sind die "Versionen" eines Images. Format: `image:tag`.

**Semantic Versioning (SemVer):** Format `MAJOR.MINOR.PATCH`, z.B. `1.4.2`.
- **MAJOR** (`X.0.0`) — Breaking Changes — was funktionierte vorher, funktioniert evtl. nicht mehr
- **MINOR** (`1.X.0`) — neue Features, abwärtskompatibel
- **PATCH** (`1.4.X`) — Bugfixes, keine API-Änderung

**`latest` vs konkrete Version:**

| Tag | Was es bedeutet | Risiko |
|---|---|---|
| `nginx:latest` | aktuellste Version, kann sich ändern | **gefährlich** — Build kann morgen anders sein als heute |
| `nginx:1.27.0` | feste Version | sicher, reproduzierbar |
| `nginx:1.27` | neueste 1.27.x — Patches werden mitgezogen | meist sicher, nimmt Bugfixes |
| `nginx:1` | neueste 1.x.x — auch Minor-Updates | etwas riskanter |
| `nginx@sha256:abc...` | exakt diese Image-Bytes | maximal sicher, aber unflexibel |

**Best Practice im Workflow:**
1. Beim Build immer **mehrere Tags** vergeben:
   ```bash
   docker build -t myapp:1.4.2 -t myapp:1.4 -t myapp:latest .
   docker push myapp:1.4.2
   docker push myapp:1.4
   docker push myapp:latest
   ```
2. In Production NIE `:latest` verwenden — immer konkrete Version pinnen
3. In Dev/Test gerne `:latest` für komfortableres Updaten

**Stolperfalle:** `latest` ist **nur ein Tag-Name**, nicht magisch. Es zeigt auf das Image, das zuletzt mit Tag `latest` gepusht wurde — nicht zwingend das chronologisch neueste!

**Praxis-Anker:** In KN-08 hast du Versionen genutzt: `leonwul/m347-kn08:frontend` (initial), `:frontend-v2` (Update-Demo), `:frontend-v3` (multistage). Beim Rolling Update hast du gezielt die Version gewechselt mit `kubectl set image deployment/frontend-deployment frontend=leonwul/m347-kn08:frontend-v2`.

### 2.3 Sicherheitsaspekte einer Registry

> **Lernziel:** "Ich kann Sicherheitsaspekte einer Registry erläutern, insbesondere den Zugriffsschutz (Authentication) und das Scannen von Images auf Sicherheitslücken (Vulnerabilities)."

**Zwei Hauptaspekte:**

**A) Zugriffsschutz / Authentication**

Wer darf ziehen / pushen?
- **Public-Registry:** jeder kann pullen, push nur mit Owner-Login
- **Private-Registry:** push UND pull nur mit Login
- Login mit `docker login registry.example.com`

**Für K8s und private Images:** `imagePullSecrets` in Pod-Spec referenziert ein Secret vom Typ `kubernetes.io/dockerconfigjson` — der Kubelet nutzt diese Credentials zum Pullen.

**B) Vulnerability Scanning**

Images enthalten oft Software mit bekannten Schwachstellen (CVEs — Common Vulnerabilities and Exposures). Beispiel: dein PHP-Image basiert auf einem Ubuntu, das eine OpenSSL-Schwachstelle hat.

**Wie Scanning funktioniert:**
1. Registry analysiert die Layer des Image
2. Vergleicht installierte Pakete + Versionen mit CVE-Datenbanken
3. Meldet Findings nach Severity (Critical / High / Medium / Low)

**Bekannte Tools:**
- **Docker Hub Scout** — eingebaute Scans auf hub.docker.com
- **Trivy** — Open-Source-Scanner (kommandozeilen-basiert)
- **GitLab Container Scanning** — bei jedem CI-Build automatisch
- **Snyk** — kommerziell

**Was tun bei Findings?**
- Base-Image aktualisieren (z.B. `FROM ubuntu:24.04` statt `FROM ubuntu:20.04`)
- Veraltete Pakete im Image entfernen oder updaten
- Minimalere Base-Images nutzen (alpine statt ubuntu, distroless statt alpine)

**Was kann gefragt werden:**
- Warum ist `:latest` ein Sicherheitsrisiko? → Du weisst nie genau welche Version läuft, kannst nicht reproduzieren wo eine Schwachstelle war
- Was ist eine CVE? → Eine standardisierte Kennung für eine bekannte Schwachstelle (z.B. CVE-2024-1234)
- Wie kann man Schwachstellen reduzieren? → Minimal-Images (alpine/distroless), regelmässige Updates, automatische Scans im CI

**Praxis-Anker:** In KN-08 hast du das Multistage-Dockerfile gebaut — die finale `nginx:alpine` Image ist viel kleiner und hat weniger Pakete = weniger Angriffsfläche als wenn du Node.js mit ins Production-Image gepackt hättest.

### 3.1 docker-compose.yml lesen + interpretieren

> **Lernziel:** "Ich kann eine docker-compose.yml Datei lesen, interpretieren und erklären."

**Grundstruktur** (alle wichtigen Felder):
```yaml
services:                              # ← Liste aller Container
  web:                                 # ← Service-Name = wird DNS-Name im Netz
    build:                             # ← Image aus Dockerfile bauen
      context: .                       # ← Build-Context
      dockerfile: Dockerfile.web       # ← welches Dockerfile (falls nicht ./Dockerfile)
    # ALTERNATIV (statt build):
    # image: nginx:1.27                # ← fertiges Image aus Registry ziehen
    container_name: my-web             # ← fester Container-Name
    ports:
      - "80:80"                        # ← Host:Container Port-Mapping
    environment:
      DB_HOST: db                      # ← ENV-Vars im Container
      DB_USER: admin
    volumes:
      - mydata:/var/lib/data           # ← named volume mount
      - ./src:/app/src                 # ← bind mount
    depends_on:
      - db                             # ← db wird zuerst gestartet
    networks:
      - my-net                         # ← in welchem Netz hängt der Container
    restart: unless-stopped            # ← Restart-Policy bei Crash
  db:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: mydb
    volumes:
      - mydata:/var/lib/mysql

networks:                              # ← Netzwerk-Definitionen
  my-net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/24

volumes:                               # ← Volume-Definitionen
  mydata:
```

**Wenn du den File interpretierst, frage dich:**
- Welche Services gibt es? (Liste aller Container)
- Welche werden gebaut (`build:`) vs welche kommen fertig aus Registry (`image:`)?
- Welche Ports gehen nach aussen? (`ports:`)
- Was sind die Abhängigkeiten? (`depends_on:`)
- Welche Daten sind persistent? (`volumes:`)
- Welche Netzwerke verbinden was?

**Was kann gefragt werden:** "Was macht dieses Compose-File?" → Du gehst Service für Service durch und erklärst wie oben.

### 3.2 Environment-Variablen — Vor- und Nachteile

> **Lernziel:** "Ich bin in der Lage die Vorteile (und Nachteile) von Environment Variables (Umgebungsvariablen) zu erklären."

**Was sind ENV-Vars?** Konfigurationswerte, die als Variablen ins Container-Environment gepusht werden. Beispiel: `DB_HOST=mysql.example.com`.

**Wie setzen?**
```yaml
services:
  app:
    image: myapp
    environment:
      DB_HOST: db                    # ← direkt im compose
      DB_PORT: 3306
    env_file:
      - .env.production              # ← aus separater Datei
```

**Vorteile:**
- ✅ **Konfiguration vom Code trennen** — gleiches Image für dev, staging, prod (12-Factor App)
- ✅ **Image bleibt wiederverwendbar** — kein Rebuild für Konfig-Änderung
- ✅ **Standard-Mechanismus** — jede Sprache liest ENV-Vars nativ (Node `process.env`, Python `os.environ`, .NET `Environment.GetEnvironmentVariable`, ...)
- ✅ **Übergabe per CLI** — `docker run -e DB_HOST=foo`

**Nachteile:**
- ❌ **Geheimnisse leaken** — wenn das Compose-File in Git landet, sind Passwörter sichtbar. Lösung: Secrets (Docker Secrets / K8s Secrets) oder env_file ohne git-tracking
- ❌ **Nicht type-safe** — alles sind Strings, leicht Fehler bei `true`/`false`/Zahlen
- ❌ **Reihenfolge der Auflösung verwirrend** — wenn ENV in mehreren Quellen (compose, env_file, Dockerfile ENV) gesetzt: welche gewinnt?
- ❌ **Build-Time vs Runtime** — React-Apps bakken `REACT_APP_*` zur Build-Zeit ins JS — ENV-Var im laufenden Container zu setzen bringt dann nichts (musste in KN-08 Aufgabe 9 mit Multistage + Substitution gelöst werden)

**Stolperfalle:** In Compose `.env`-Files: Werden für die Variablen-Substitution **im Compose-File selbst** verwendet (`${VAR}`), NICHT zwingend in Container injiziert. Für das musst du `env_file:` setzen.

**Praxis-Anker:** In KN-08 hast du gelernt warum dynamic-ENV für React tricky ist — Build-Time-Substitution macht ENV-Vars im fertigen Image-JS hardcoded.

### 3.3 depends_on + seine Einschränkungen

> **Lernziel:** "Ich kann Abhängigkeiten zwischen Diensten erklären (z. B. depends_on), um sicherzustellen, dass die Datenbank gestartet wurde, bevor die Applikation startet, aber kann auch die Einschränkung von depends_on erklären."

**Was `depends_on` macht:**
```yaml
services:
  web:
    image: myapp
    depends_on:
      - db
  db:
    image: mysql
```
→ `db` wird **gestartet** bevor `web` startet. Auch beim `docker compose down` wird `web` zuerst gestoppt.

**Die GROSSE Einschränkung:** `depends_on` heisst nur **"db startet zuerst"** — NICHT **"db ist bereit"**!

**Was passiert in der Realität:**
1. `db`-Container startet (Process läuft)
2. MySQL braucht ~10 Sek zum Booten + DB-Initialisierung
3. `web`-Container startet (depends_on erfüllt → web läuft sofort)
4. `web` versucht zu `db:3306` zu verbinden → ABER MySQL hört noch nicht → **Connection refused**

**Lösungen für "warte bis wirklich bereit":**

**Option A — Healthcheck mit condition:**
```yaml
services:
  db:
    image: mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping"]
      interval: 5s
      retries: 10
  web:
    image: myapp
    depends_on:
      db:
        condition: service_healthy     # ← wartet auf gesunden Status
```

**Option B — Retry-Logik in der App:**
- App muss selbst handeln: DB-Verbindung in einer Schleife versuchen, mit Backoff retry-en
- Robuster, weil DB im laufenden Betrieb auch mal weg sein kann (Restart, Crash)

**Option C — wait-for-it Skripte:**
- Shell-Skript das in Loop pingt, wartet bis Port hört, dann erst die App startet
- Wird im Container als ENTRYPOINT gestartet

**Was kann gefragt werden:** "Du hast `depends_on: db` gesetzt aber die App crasht beim Start mit 'Connection refused'. Warum?" → Weil depends_on nur startet, nicht wartet auf Bereitschaft.

**Praxis-Anker:** In KN-04 / KN-07 hattest du auch ohne Healthcheck konfiguriert — meistens reicht's, weil das Web-Image so lange zum Starten braucht dass die DB schon up ist. Im Crashfall: einfach Page neu laden.

### 3.4 Lebenszyklus: up, down, logs

> **Lernziel:** "Ich kann den Lebenszyklus einer Multi-Container-Umgebung mit Befehlen wie docker-compose up -d, down und logs -f effizient steuern."

**Die wichtigsten Befehle:**

```bash
# STARTEN
docker compose up                # vordergrund, sieht alle Logs, Strg+C = stop
docker compose up -d             # detached, im Hintergrund (production-style)
docker compose up -d --build     # vorher neu bauen (für lokale Dockerfile-Änderungen)
docker compose up -d web db      # nur bestimmte Services hochfahren

# STATUS
docker compose ps                # Liste aller Container im Stack
docker compose ps -a             # auch gestoppte

# LOGS
docker compose logs              # alle Logs aller Container
docker compose logs web          # nur ein Service
docker compose logs -f           # follow (live anhängen)
docker compose logs --tail=100   # letzte 100 Zeilen

# STOPPEN / NEUSTARTEN
docker compose stop              # alle Container stoppen (Container bleibt)
docker compose start             # gestoppte wieder starten
docker compose restart           # stop + start
docker compose restart web       # nur einen Service

# AUFRÄUMEN
docker compose down              # Container stoppen UND löschen, Netzwerke entfernen
docker compose down -v           # zusätzlich Volumes löschen (DATEN WEG!)
docker compose down --rmi all    # zusätzlich Images löschen

# IN LAUFENDEM CONTAINER
docker compose exec web bash     # Shell im laufenden web-Container

# NEU BAUEN OHNE CACHE
docker compose build --no-cache web

# ANDERE COMPOSE-DATEI
docker compose -f docker-compose.prod.yml up -d
```

**Was passiert bei `up -d`?**
1. Build (falls `build:` definiert und Image fehlt oder `--build` gesetzt)
2. Pull (falls `image:` definiert und nicht lokal)
3. Netzwerke anlegen (falls noch nicht da)
4. Volumes anlegen (falls noch nicht da)
5. Container erstellen (mit Name, Netz, ENV, Mounts)
6. Container starten
7. (ohne `-d`: an Logs anhängen)

**Was kann gefragt werden:**
- Unterschied `down` vs `stop`? → stop pausiert (Container bleibt), down löscht alles
- `down -v` — was wird zusätzlich gelöscht? → die Volumes (Achtung: DB-Daten weg!)
- Wie sehe ich die Logs eines bestimmten Containers live mit? → `docker compose logs -f <service>`

**Praxis-Anker:** Beim lokalen Test in KN-08 hattest du den ganzen Stack mit `docker compose up -d` gestartet, `docker compose ps` zur Verifikation, dann am Ende `docker compose down`.

## Block 4 — Kubernetes

### 4.1 Architektur + API-Objekte

> **Lernziel:** "Ich kann die Architektur eines Kubernetes-Clusters (Control Plane, Worker Nodes, Kubelet) und die Funktion der API-Objekte (Pods, Deployments, Services, ConfigMaps, Secrets) erklären."

**Architektur — Visualisiert:**

```
        ┌─────────────────────────── Kubernetes Cluster ─────────────────────────┐
        │                                                                          │
        │   ┌─────────────────── Control Plane (Master Node) ─────────────────┐  │
        │   │                                                                    │  │
        │   │   API Server  ──  etcd  ──  Scheduler  ──  Controller Manager    │  │
        │   │       ↑                                                            │  │
        │   └───────┼────────────────────────────────────────────────────────────┘  │
        │           │ HTTPS                                                          │
        │   ┌───────┼────────────┐  ┌────────────────────┐  ┌───────────────────┐  │
        │   │       ▼            │  │                    │  │                   │  │
        │   │   Worker Node 1    │  │   Worker Node 2    │  │   Worker Node 3   │  │
        │   │                    │  │                    │  │                   │  │
        │   │ Kubelet            │  │ Kubelet            │  │ Kubelet           │  │
        │   │ Kube-Proxy         │  │ Kube-Proxy         │  │ Kube-Proxy        │  │
        │   │ Container Runtime  │  │ Container Runtime  │  │ Container Runtime │  │
        │   │ ┌────┐ ┌────┐      │  │ ┌────┐ ┌────┐      │  │ ┌────┐            │  │
        │   │ │Pod │ │Pod │      │  │ │Pod │ │Pod │      │  │ │Pod │            │  │
        │   │ └────┘ └────┘      │  │ └────┘ └────┘      │  │ └────┘            │  │
        │   └────────────────────┘  └────────────────────┘  └───────────────────┘  │
        └──────────────────────────────────────────────────────────────────────────┘
                                          ▲
                                          │ kubectl / Dashboard / API
                                       User/Admin
```

**Control Plane (Master) — verwaltet den Cluster:**
- **API Server** — zentrale Anlaufstelle für alle kubectl/HTTPS-Anfragen
- **etcd** — verteilte Key-Value-DB, speichert den kompletten Cluster-State
- **Scheduler** — entscheidet auf welchem Node ein neuer Pod laufen soll
- **Controller Manager** — überwacht Soll/Ist (z.B. ReplicaSet: "soll 3 Pods, sind nur 2 → starte einen")

**Worker Nodes — führen die Workloads aus:**
- **Kubelet** — der Agent auf jedem Node; spricht mit dem API-Server und sorgt dafür dass die Pods laufen
- **Kube-Proxy** — verwaltet Netzwerk-Regeln auf dem Node (für Services)
- **Container Runtime** — startet die Container (containerd, Docker, CRI-O)

**Die wichtigsten API-Objekte:**

| Objekt | Was es ist | Wofür |
|---|---|---|
| **Pod** | kleinste deploybare Einheit, 1+ Container die sich Netz/Volumes teilen | Laufzeit-Einheit |
| **ReplicaSet** | sorgt für N gleiche Pods | wird meist von Deployment verwaltet |
| **Deployment** | verwaltet ReplicaSet + Rolling Updates + Rollback | Standard für stateless Workloads (Web-Apps, APIs) |
| **StatefulSet** | wie Deployment, aber mit stabiler Identität pro Pod (Pod-0, Pod-1...) + eigenem Volume pro Pod | Datenbanken, verteilte Systeme mit Identität |
| **Service** | stabile Netzwerk-Adresse + Load-Balancing zu einer Gruppe von Pods | Pods sind ephemer, Services nicht |
| **ConfigMap** | Key-Value Konfig (nicht-geheim, im Klartext) | Konfig (URLs, Feature-Flags) |
| **Secret** | Key-Value mit base64-Encoding | Passwörter, API-Keys |
| **Ingress** | HTTP-Reverse-Proxy für mehrere Services (Hostname/Pfad-Routing + TLS) | EIN Eingangspunkt für viele Apps |

**Was kann gefragt werden:**
- Welche Komponenten gehören zur Control Plane? → API Server, etcd, Scheduler, Controller Manager
- Was läuft auf einem Worker Node? → Kubelet, Kube-Proxy, Container Runtime, die Pods
- Wo wird der Cluster-State persistent gespeichert? → etcd
- Was ist die kleinste deploybare Einheit? → Pod (nicht Container!)
- Unterschied Deployment vs StatefulSet? → Deployment für stateless (Pods austauschbar), StatefulSet für stateful (stabile Identität + eigene Volumes)
- Wofür ConfigMap, wofür Secret? → ConfigMap für nicht-geheime Konfig, Secret für Passwörter etc.

- KN-07: deine erste App mit Deployment + Service + ConfigMap + Secret
- KN-08: alle Konzepte zusammen — 4 Deployments, 4 Services, ConfigMap, Secret, Rolling Update via Deployment

### 4.2 YAML-Manifeste interpretieren

> **Lernziel:** "Ich bin fähig, vorgegebene YAML-Manifesten zu interpretieren und zu erklären."

**Grundstruktur jedes K8s-Objekts (4 Top-Level-Felder):**
```yaml
apiVersion: apps/v1     # ← welche API-Version (Pod = v1, Deployment = apps/v1)
kind: Deployment        # ← welcher Objekt-Typ
metadata:               # ← Name + Labels
  name: my-app
  labels:
    app: my-app
spec:                   # ← die "gewünschte" Konfiguration
  ...
```

**Beispiel-Deployment vollständig durchgegangen:**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp-deployment        # ← Name des Deployments
  labels:
    app: webapp
spec:
  replicas: 3                    # ← genau 3 gleiche Pods sollen laufen
  selector:                      # ← welche Pods gehören zu mir?
    matchLabels:
      app: webapp                # ← Pods mit Label app=webapp
  template:                      # ← BLUEPRINT für die Pods
    metadata:
      labels:
        app: webapp              # ← Label muss zum Selector passen!
    spec:                        # ← Pod-Spec
      containers:
      - name: webapp
        image: leonwul/myapp:v1
        ports:
        - containerPort: 8080    # ← welchen Port hört der Container
        env:                     # ← ENV-Vars
        - name: DB_URL
          valueFrom:
            configMapKeyRef:
              name: m347-config  # ← aus welcher ConfigMap
              key: DB_URL        # ← welcher Key dort
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: m347-secret  # ← aus welchem Secret
              key: DB_PASSWORD
```

Ein **Service** dazu:
```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-service           # ← DNS-Name im Cluster
spec:
  type: NodePort
  selector:
    app: webapp                  # ← routet zu Pods mit Label app=webapp
  ports:
  - port: 80                     # ← der Service-Port
    targetPort: 8080             # ← der Pod-Port
    nodePort: 30100              # ← extern auf jedem Node erreichbar
```

**Wie lese ich ein YAML-Manifest?**
1. **Welcher kind?** Verstehe was für ein Objekt das ist.
2. **Was ist im spec?** Was ist die gewünschte Konfiguration?
3. **Bei Service:** wohin routet er? (selector → Pods mit passenden Labels)
4. **Bei Deployment:** wie viele replicas? Welches Image? Welche ENV-Vars?
5. **Bei ConfigMap/Secret:** welche Keys+Values?
6. **Welche anderen Objekte referenziert es?** (ConfigMapKeyRef, SecretKeyRef, Service-Selector → Pod-Labels)

**Was kann gefragt werden:** "Erklär mir was dieses YAML macht." Du gehst Top-Level-Felder durch (apiVersion → kind → metadata → spec) und erklärst Sektion für Sektion.

**Praxis-Anker:** Du hast viele YAMLs geschrieben/gelesen in KN-07 + KN-08. In KN-08 hast du 7 verschiedene YAMLs (ConfigMap, Secret, 4× Deployment+Service, LoadBalancer).

### 4.3 Service-Typen: ClusterIP, NodePort, LoadBalancer

> **Lernziel:** "Ich kann verschiedene Service-Typen (ClusterIP, NodePort, LoadBalancer) erklären und zu definieren, welche Services, welchen Typ verwenden sollte."

**Service** = stabile Netzwerk-Adresse vor einer Gruppe von Pods. Pod-IPs ändern sich (Pods sind ephemer), Service-IP nicht.

**Die 3 Haupt-Typen:**

| Typ | Wo erreichbar | Use-Case | Beispiel |
|---|---|---|---|
| **ClusterIP** (default) | nur **innerhalb** des Clusters | interne Services (DB, internes API) | Mongo-Service, Account-zu-Account-Communication |
| **NodePort** | **extern** über `<Node-IP>:<NodePort>` (30000-32767) | einfacher externer Zugang, kein Cloud-LB nötig | Demo-Apps, Dev-/Test-Umgebungen |
| **LoadBalancer** | **extern** über einen Cloud-Load-Balancer mit fester IP/DNS | Production externe Apps in der Cloud | Frontend einer Production-App |

**ClusterIP-Beispiel (Default):**
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mongo-service
spec:
  selector:
    app: mongo
  ports:
  - port: 27017
    targetPort: 27017
# kein type angegeben → ClusterIP
```
Nur Pods im Cluster können `mongo-service:27017` aufrufen. Aus dem Internet → nicht erreichbar.

**NodePort-Beispiel:**
```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-service
spec:
  type: NodePort
  selector:
    app: webapp
  ports:
  - port: 80                 # Service-internal
    targetPort: 8080         # Container-Port
    nodePort: 30100          # Extern: <Node-IP>:30100
```
Von ausserhalb erreichbar über **jede** Node-IP des Clusters auf Port 30100.

**LoadBalancer-Beispiel:**
```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-service
spec:
  type: LoadBalancer
  selector:
    app: webapp
  ports:
  - port: 80
    targetPort: 8080
```
K8s auf einer Cloud (EKS, GKE, AKS) provisioniert automatisch einen Cloud-LB (AWS ELB, GCP LB, Azure LB) mit fester IP/DNS — der LB routet zum NodePort dahinter.

**Entscheidungs-Hilfe:**
- "Soll der Service nur intern erreichbar sein?" → **ClusterIP**
- "Brauche ich externen Zugang aber bin nicht in der Cloud (oder die Cloud-LB ist zu teuer)?" → **NodePort**
- "Production in der Cloud, brauche stabilen externen Endpunkt mit Health-Checks?" → **LoadBalancer**

**Stolperfalle bei LoadBalancer:** Nur funktional in einem Cloud-K8s mit Cloud-Controller (EKS, GKE, AKS). In MicroK8s/Minikube bleibt `EXTERNAL-IP: <pending>` weil kein Cloud-Controller drin ist.

**Was kann gefragt werden:**
- Welcher Typ ist Default? → ClusterIP
- Welcher Port-Range bei NodePort? → 30000-32767
- Warum funktioniert LoadBalancer in MicroK8s nicht "out of the box"? → kein Cloud-Controller → keine Cloud-LB-Provisionierung möglich

**Praxis-Anker:**

- KN-08: alle 4 Services als NodePort + zusätzlich externer AWS NLB für Frontend (manuell konfiguriert, weil MicroK8s den `type: LoadBalancer` nicht selbst provisioniert)

## Selbsttest — mögliche Prüfungsfragen

### Docker
1. **Was ist der Unterschied zwischen einer VM und einem Container? Nenne 3 Vorteile von Containern.**
   → VM hat eigenen Kernel, Container teilt sich den Host-Kernel. Vorteile: kleiner, schneller Start, weniger Ressourcen.

2. **Was ist der Unterschied zwischen einem Image und einem Container? Analogie?**
   → Image = Bauplan/Klasse, Container = Instanz/Objekt. Aus einem Image können viele Container gestartet werden.

3. **Was machen die Dockerfile-Anweisungen FROM, COPY, WORKDIR, RUN, EXPOSE, ENV, CMD?**
   → FROM=Basis-Image, COPY=Dateien rein, WORKDIR=Arbeitsverzeichnis setzen, RUN=Build-Zeit-Befehl, EXPOSE=Port dokumentieren, ENV=Umgebungsvariable, CMD=Start-Befehl.

4. **Öffnet `EXPOSE 80` im Dockerfile wirklich den Port?**
   → Nein, nur Dokumentation. Tatsächlich öffnen mit `docker run -p ...`.

5. **Was ist eine Layer? Wann wird sie gecached?**
   → Eine Layer ist eine Filesystem-Schicht — eine pro Dockerfile-Anweisung. Cached wenn die Anweisung + Input-Files unverändert sind. Reihenfolge im Dockerfile beeinflusst Cache-Effizienz.

6. **3 Arten von Volumes — wo werden die Daten gespeichert?**
   → bind mount = beliebiger Host-Pfad, volume = Docker-managed, tmpfs = RAM.

7. **Warum sollte man eine user-defined bridge dem default-bridge vorziehen?**
   → DNS-Auflösung via Container-Name funktioniert, bessere Isolation, flexibler.

### Container Registry
8. **Was ist eine Container Registry und wofür braucht man sie?**
   → Zentraler Speicher für Images, ermöglicht Verteilung + Versionierung + Reproduzierbarkeit.

9. **`:latest` vs `:1.4.2` — was ist sicherer und warum?**
   → `:1.4.2` ist sicherer, weil reproduzierbar. `:latest` kann sich ändern → Build von heute ≠ Build von morgen.

10. **Wie schützt man eine Registry und ihre Inhalte?**
    → Authentication (Login für push/pull), Vulnerability Scanning (CVE-Checks), minimal-Base-Images, regelmässige Updates.

### Docker Compose
11. **Was macht `depends_on` und was macht es NICHT?**
    → Macht: Startreihenfolge (db zuerst, web danach). Macht NICHT: warten bis db bereit ist.

12. **Vor- und Nachteile von Environment-Variablen für Konfiguration?**
    → Vor: Konfig vom Code getrennt, Image wiederverwendbar. Nach: Geheimnisse können leaken, alles strings, Build-Time-Werte können nicht via ENV gesetzt werden (React-Beispiel).

13. **Unterschied `docker compose down` vs `docker compose stop`?**
    → stop = pausiert Container; down = stoppt + löscht Container + Netze (mit `-v` auch Volumes).

### Kubernetes
14. **Welche Komponenten gehören zur Control Plane?**
    → API Server, etcd, Scheduler, Controller Manager.

15. **Was läuft auf einem Worker Node?**
    → Kubelet, Kube-Proxy, Container Runtime, die Pods.

16. **Was ist die kleinste deploybare Einheit in K8s?**
    → Pod (nicht Container — ein Pod kann mehrere Container haben).

17. **Wozu braucht man einen Service, wenn man Pods direkt ansprechen könnte?**
    → Pod-IPs ändern sich (ephemer); Service hat stabile Adresse + Load-Balancing.

18. **Welche Service-Typen gibt es und wann nutzt man welchen?**
    → ClusterIP (intern, default), NodePort (extern via Node-IP+Port), LoadBalancer (Cloud-LB davor, Production).

19. **Wofür ConfigMap vs Secret?**
    → ConfigMap für nicht-geheime Konfig (URLs, Flags); Secret für Geheimnisse (Passwörter, API-Keys) — base64-encoded aber NICHT verschlüsselt!

20. **Wie macht K8s ein Rolling Update?**
    → Standard `RollingUpdate`-Strategie eines Deployments: alte Pods werden sukzessive durch neue ersetzt; immer mindestens N-maxUnavailable Pods bleiben Ready → Zero Downtime.
